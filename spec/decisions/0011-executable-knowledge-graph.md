# Decision: An executable knowledge graph over the run substrate

**Status:** Approved
**Date:** 2026-07-23
**Owner:** alex
**Tags:** product, domain-model, knowledge-graph, positioning
**Source Idea:** executable-knowledge-platform
**Supersedes:** —
**Superseded By:** —

## Context

The founder repositioned Chatwright as an executable knowledge platform for
conversational UX (idea:
[executable-knowledge-platform](../ideas/executable-knowledge-platform.md)),
naming many surfaces: jobs, recipes, patterns, capabilities, implementations,
demos, code examples, open-source bots, compatibility data, tests,
recordings, community. Treating each as its own product would multiply
formats, stores and UIs. A domain model is needed that powers documentation,
demos, runtime, search, tests, repository metadata and community
contributions from a small set of abstractions.

## Decision

Chatwright is modelled as **two layers**.

### Layer 1 — the executable substrate (exists today)

A **Runtime** executes a **Scenario** against one or more **Bots** over an
emulated **Platform** surface, journals everything, and persists the result
as a **Recording**. There is exactly one Recording format: the run bundle
(`https://chatwright.dev/formats/run-bundle/v1`). A live demo is a run
without assertions; a test is a run with assertions; a benchmark cell is a
run driven by an AI actor. Nothing in this decision adds execution machinery
or a second recording format.

### Layer 2 — the knowledge graph (new)

Descriptive nodes whose edges point into the substrate:

- **Job** — an intent someone wants to accomplish ("collect RSVP", "split a
  restaurant bill"). Deliberately thin: title, description, related Jobs,
  and `solved-by` edges to Recipes. Job pages are mostly generated views.
- **Recipe** — the central content asset: an executable answer to one or
  more Jobs. Owns prose (conversation design, trade-off discussion),
  references one or more Implementations, and names a canonical demo
  (Bot + Scenario) plus supporting evidence (Recordings, screenshots,
  tests). Platform-neutral in its problem framing.
- **Capability** — a platform primitive identified by a **capability key**:
  a stable dotted path in one capability tree (for example
  `messaging.buttons.inline`, `messaging.message.edit`,
  `messaging.payments`). A Capability has two facets: machine-readable
  compatibility data (per-platform support statements — the
  browser-compat-data shape) and a prose page largely generated by joining
  the Capability to the Implementations that use it, answering "how do I
  solve this on each platform", never merely "is it supported".
- **Implementation** — one Recipe realised on one Platform with one
  technique ("RSVP via Telegram inline buttons"). Carries a tier
  (**official / alternative / community** — the MDN model; never a single
  recommended answer), documented advantages and disadvantages, `uses`
  edges to Capabilities, a source Repository, a demo Bot and evidence
  Recordings. The unit of comparison.
- **Bot** — a runnable black box addressable by the Runtime: an iframe URL
  (postMessage transport) or an HTTPS endpoint (webhook/API transport),
  speaking platform-native payloads (decision
  [0012](0012-black-box-bot-protocol.md)). Implementation language is
  invisible by design. One Bot may serve many Implementations.
- **Platform** — a messaging platform as Chatwright represents it: an
  emulator with declared capability fidelity, compatibility data keyed by
  capability keys, and its native wire vocabulary.
- **Author** and **Repository** — attribution and packaging: a GitHub-
  anchored identity, and a git repository carrying a `CHATWRIGHT.md`
  manifest (decision [0013](0013-chatwright-md-federation.md)).

**Scenario** and **Recording** remain substrate terms with their existing
glossary meanings; a Conversation stays a view of a run's journal, not a
stored entity.

## Rationale

The model buys five unifications:

1. **Recording = run bundle.** Demos, tests, benchmarks, live captures and
   community show-and-tell all persist and replay through one format, one
   player.
2. **One capability vocabulary everywhere.** The same capability keys drive
   compatibility tables, emulator fidelity declarations (extending the
   existing fidelity-labels concept and the capability profiles of
   idea:third-party-emulator-backends), `CHATWRIGHT.md` declarations, bot
   protocol negotiation and search facets. No parallel taxonomies.
3. **Compatibility is data, not prose.** Support statements live in small
   per-capability data files (browser-compat-data's proven shape: status,
   spec link, per-platform `since`/`partial`/`notes`); pages render from
   data. A caniuse-style flat view is derived, never authored.
4. **Community mechanics generalise.** Stars, adoption counts, clone
   buttons and leaderboards specified for arena scenarios
   (idea:actor-model-arena) apply uniformly to knowledge-graph nodes
   (decision [0014](0014-community-metrics.md)).
5. **Search is graph traversal.** Searching by job, recipe, capability,
   platform, framework, language, repository, author, bot or tag is one
   index over typed nodes and edges, not per-surface search engines.

## Declined Alternatives

### One node type with tags

Rejected: Jobs, Recipes and Capabilities have different lifecycles,
ownership and machine-readable facets; flattening everything into a
"document" reproduces the wiki-decay this platform exists to beat.

### Recipe and Implementation merged

Rejected: the compare experience ("inspect trade-offs") requires several
Implementations per Recipe as peers; merging forces one implementation to
be "the" answer.

### Job and Recipe merged

Rejected: the relation is many-to-many; a Job is an entry point, a Recipe
is an asset.

### A separate benchmark/community product

Rejected: the arena idea already defines these mechanics on the same
substrate; duplication would split identity and metrics.

## Consequences at Decision Time

- The knowledge graph needs a home and a contribution surface: the curated
  `chatwright/recipes` repository (decision 0013) holds first-party Jobs,
  Recipes, Capabilities (including compatibility data) and the federated
  registry.
- Recipes and Implementations must name demo Bots and Scenarios by stable
  reference, which requires the bot manifest (0013) and, eventually, a
  portable scenario format (open research).
- Generated pages (Job views, capability tables, implementation
  comparisons) become a rendering concern of the website, not new content
  types.
- The glossary gains the knowledge-graph terms; existing substrate terms
  are unchanged.
